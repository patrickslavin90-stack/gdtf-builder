## What It Does

1. **User uploads** a file (MA3 patch XML, DMX chart image/PDF, CSV, text) or types a channel list manually
2. **Frontend** sends fixture metadata + channel data to the Netlify Function
3. **Function** calls Gemini 2.5 Flash with a strict GDTF 1.2 system prompt
4. **Post-processing** fixes known Gemini output bugs (namespaces, empty offsets, bad geometry refs, etc.)
5. **Frontend** validates the XML, shows a preview, and packages it as a `.gdtf` ZIP for download
6. User drops the `.gdtf` into MA3/GDTF Share and patches

**No API key is exposed to the user.** The Gemini key lives in Netlify's environment variables server-side only.

## Architecture

```
Browser (index.html)
    │
    │  POST /api/generate
    │  { manufacturer, fixtureName, dmxMode, channels, ... }
    ▼
Netlify Function (generate.js)         ← GEMINI_API_KEY lives here only
    │
    │  POST generativelanguage.googleapis.com
    │  model: gemini-2.5-flash
    ▼
Gemini API
    │
    │  Raw XML response
    ▼
Post-Processing Pipeline
    ├── stripNamespaces()      remove xmlns/xsi attrs
    ├── fixManufacturer()      child element → FixtureType attribute
    ├── fixGeometryRefs()      validate + correct geometry references
    └── assignOffsets()        sequential 1,2,3... per DMXMode
    │
    │  { xml: "<?xml..." }
    ▼
Browser — applyMA3Fixes() client-side + validate() + download as .gdtf ZIP
```

### Why Netlify Functions (not Edge Functions)

The original implementation used `export default async (req) =>` + `export const config` — **Netlify Edge Function** syntax. Edge functions have a 50ms CPU limit and run in a V8 isolate. Gemini calls take 5–15 seconds. This caused Netlify to return an HTML error page instead of JSON, producing the `"Unexpected token '<'"` error in the frontend.

The correct format is standard **Netlify Functions** using `exports.handler = async function(event, context)` which run in Node.js with a 10-second timeout and full network access.

## GDTF Format — What We Learned

GDTF (General Device Type Format) is a ZIP file containing a `description.xml`. The XML file **must** be named exactly `description.xml` inside the ZIP — our first iteration stored it under the source filename which caused every GDTF Share upload to fail with a "node error".

### Key structural facts

| Element | Notes |
|---|---|
| `<GDTF DataVersion="1.2">` | Root element. No `xmlns` attributes — they break browser `DOMParser` |
| `<FixtureType>` | All identity fields are **attributes**, not child elements |
| `Manufacturer` | **Attribute on FixtureType**, not `<Manufacturer Name="x"/>` |
| `FixtureTypeID` | Must be uppercase hex GUID: `XXXXXXXX-XXXX-XXXX-XXXX-XXXXXXXXXXXX` |
| `RefFT` | Omit entirely — empty string `""` fails XSD `guidtype` validation |
| `DMXChannel Offset` | Sequential integers `"1"`, `"2"`, `"3"` — 16-bit: `"15,16"`. Empty = -1 footprint in MA3 |
| `InitialFunction` | Format: `"GeometryName_AttributeName.AttributeName.CFName"` |
| `ChannelFunction Default` | Must fall within that specific function's DMX range, not the parent channel default |
| `Wheel=` on ChannelFunction | Required on any CF whose Attribute references a wheel (Color, Gobo, Prism) |

### The ZIP packaging rule

```javascript
// CORRECT — always name it description.xml inside the ZIP
zip.file('description.xml', generatedXML);

// WRONG — keeps the source filename
zip.file('my-fixture.xml', generatedXML);  // ← causes "node error" on GDTF Share
```

## MA3 Compliance Rules

Rules discovered through iteration against real MA3 imports and the official GDTF XSD + Schematron files at [github.com/mvrdevelopment/tools](https://github.com/mvrdevelopment/tools):

| Rule | Detail |
|---|---|
| `DataVersion="1.2"` | MA3 3.x requires 1.2, not 1.0 |
| `InitialFunction` on all DMXChannels | Without it MA3 shows "Other / 0 modes" |
| `Default=` on every ChannelFunction | Must be within that CF's own range |
| `CustomName=""` on every ChannelFunction | Required attribute |
| `Max="1.000000" Min="0.000000"` on every CF | Required attributes |
| `RealAcceleration="0.000000"` on every CF | Required attribute |
| `Wheel=` on wheel-linked CFs | Causes "No Linked Wheel Defined" without it |
| `Feature="Color.RGB"` for ColorAdd_R/G/B | `Color.Color` is wrong — MA3 can't identify RGB |
| `PhysicalUnit="ColorComponent"` on RGB | Required for MA3 colour picker |
| `<Feature Name="RGB"/>` in Color FeatureGroup | Must exist alongside `Feature="Color.RGB"` |
| No periods in ChannelSet names | `"Beam2-5deg"` not `"Beam 2.5°"` |
| `DMXMode Description=""` | Required attribute |
| `CanHaveChildren="Yes"` | Required on FixtureType |
| `ThumbnailOffsetX/Y="0"` | Required on FixtureType |
| Axis with no `Model` attribute | Pure rotation axes must not have a Model |
| `SVGOffsetX/Y` etc on all Models | 6 SVG offset attributes required |
| `RectangleRatio` + `ThrowRatio` on Beam | Required geometry attributes |
| `<Revisions>`, `<FTPresets/>`, `<Protocols/>` | Required end-of-FixtureType blocks |
| Full `<PhysicalDescriptions>` children | `AdditionalColorSpaces`, `Gamuts`, `Connectors`, `Properties` |
| `<ColorSpace Mode="sRGB" Name=""/>` | Name="" attribute required |
| `Master="Grand"` on Dimmer LogicalChannel | Hooks into grandmaster fader |
| `TiltRotate` for tilt speed channels | Not `Control1` |
| Geometry refs in DMXChannels must exist | XSD enforces referential integrity |
| DMXMode `Geometry=` must be top-level | Can't reference nested geometry |

## Known Gemini Output Bugs & Fixes

These are issues Gemini 2.5 Flash produces consistently, regardless of prompt instructions. All are caught and corrected server-side in `generate.js`:

### 1. XML Namespace Declaration
**Gemini output:**
```xml
<GDTF xmlns="http://schemas.gdtf.eu/gdtf/1.2/gdtf.xsd" 
      xmlns:xsi="..." DataVersion="1.2">
```
**Problem:** Browser `DOMParser` throws `"The string did not match the expected pattern"` when trying to find elements without namespace prefix.  
**Fix:** `stripNamespaces()` — regex strips all `xmlns` and `xsi:` attributes before the response leaves the server.

### 2. Empty Offset Attributes
**Gemini output:**
```xml
<DMXChannel Offset="" Geometry="Head" ...>
```
**Problem:** MA3 calculates DMX footprint as -1, fixture cannot be patched.  
**Fix:** `assignOffsets()` — processes each DMXMode block and assigns sequential integers (1, 2, 3...) to any channel with a blank offset.

### 3. Manufacturer as Child Element
**Gemini output:**
```xml
<FixtureType Name="MyFixture" ...>
  <Manufacturer Name="My Co"/>   ← WRONG
```
**Problem:** MA3 can't find the manufacturer name; fixture imports with no identity.  
**Fix:** `fixManufacturer()` — detects the child element, extracts the value, removes the element, adds `Manufacturer="My Co"` as an attribute on `FixtureType`.

### 4. Phantom Geometry References
**Gemini output:**
```xml
<DMXChannel Geometry="Fixture_Beam" ...>
<!-- but Geometries tree only has: Base, Head, Beam -->
```
**Problem:** Gemini confuses the `InitialFunction` naming format (`Geo_Attr`) with the actual geometry `Name`. The reference fails XSD key validation.  
**Fix:** `fixGeometryRefs()` — cross-references all geometry names defined in the tree vs used in channels, strips underscore suffixes from invalid refs, falls back to the nearest matching or top-level geometry.

### 5. Duplicate FixtureTypeID
**Problem:** Gemini sometimes reuses the same hardcoded GUID across different generations.  
**Fix:** System prompt Rule 11 explicitly instructs Gemini to generate a unique random GUID each time. The frontend `makeGUID()` also regenerates if the GUID fails the uppercase-hex regex.

### 6. Wrong model API names (timeline)
| Date | Model used | Result |
|---|---|---|
| Initial | `claude-sonnet-4-20250514` | Working (Anthropic) |
| First Gemini attempt | `gemini-1.5-flash` | 404 — deprecated |
| Second attempt | `gemini-2.0-flash` | 404 — not available to new users |
| Third attempt | `gemini-2.5-flash-preview-04-17` | 404 — preview expired |
| Current | `gemini-2.5-flash` | ✅ Working |

## Post-Processing Pipeline

Every Gemini response passes through this chain server-side before being returned to the browser:

```javascript
// 1. Strip markdown wrapping
xml = xml.replace(/```xml\n?/gi, '').replace(/```\n?/g, '').trim();

// 2. Extract GDTF block (handles preamble text Gemini sometimes adds)
const xs = xml.indexOf('<?xml'), xe = xml.lastIndexOf('</GDTF>');
xml = xml.slice(xs, xe + 7);

// 3. Strip namespace declarations (breaks browser DOMParser)
xml = stripNamespaces(xml);

// 4. Move Manufacturer from child element to FixtureType attribute
xml = fixManufacturer(xml);

// 5. Validate and correct geometry references
xml = fixGeometryRefs(xml);

// 6. Assign sequential DMX offset numbers to any empty Offset attributes
xml = assignOffsets(xml);
```

The frontend also runs `applyMA3Fixes()` as a second pass, which handles:
- `DataVersion` upgrade to 1.2
- `CanHaveChildren`, `ThumbnailOffsetX/Y` defaults
- `CustomName`, `Max`, `Min`, `RealAcceleration` on all ChannelFunctions
- `Default` set to `DMXFrom` value if missing
- `InitialFunction` generation if missing
- `Feature="Color.RGB"` correction on ColorAdd attributes
- SVG offset attributes on Models
- `RectangleRatio`/`ThrowRatio` on Beam geometry
- Missing `Revisions`, `FTPresets`, `Protocols` blocks
- Period removal from ChannelSet names

## Deploying to Netlify

The project uses **Netlify Drop-to-deploy** (git-connected).

**Auto-deploy:** Any push to `main` branch triggers a deploy automatically.

**Manual deploy from terminal:**
```powershell
cd D:\Dev\gdtf-builder
git add .
git commit -m "your message"
git push origin master:main   # note: local branch is master, remote is main
```

**Note on branch naming:** The repo was initialised locally with `git init` which creates `master`. GitHub created `main`. Always push as `master:main`.

### netlify.toml

```toml
[build]
  publish = "."
  functions = "netlify/functions"

[[redirects]]
  from = "/api/*"
  to = "/.netlify/functions/:splat"
  status = 200
```

This routes `POST /api/generate` → `/.netlify/functions/generate` → `netlify/functions/generate.js`.

## Validation Checklist

Before pushing a new `generate.js` to production, run the local test and verify all of these pass:

```
✓ Has GDTF root
✓ DataVersion 1.2
✓ Has FixtureType
✓ Manufacturer as attribute (not child element)
✓ LongName and ShortName attributes present
✓ No xmlns namespace declaration
✓ Has DMXMode
✓ Has DMXChannels
✓ No empty Offsets
✓ All geometry refs valid (used = defined)
✓ Offsets match channel count
✓ Has Revisions block
✓ GUID is uppercase hex
✓ No periods in ChannelSet names
✓ ColorAdd_R/G/B use Feature="Color.RGB"
✓ PhysicalDescriptions complete
✓ ColorSpace has Name="" attribute
✓ File packages as description.xml inside ZIP
```

Also validate at [GDTF Share](https://gdtf-share.com) before considering a fixture "done" — upload and check for node errors or warnings in the validator.
